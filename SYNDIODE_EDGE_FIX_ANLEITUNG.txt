================================================================================
SYNDIODE.COM — WARUM EINIGE PFADE 404 LIEFERN & WAS DU WO ÄNDERN MUSST
================================================================================
Stand: Anleitung für Betreiber. Keine Secrets in diese Datei schreiben.

Empfohlene öffentliche Basis: **https://syndiode.com** (ohne /nomad), damit Edge-
Whitelists nicht für jeden neuen Endpunkt nachgezogen werden müssen. Ein
Pfad-Prefix /nomad ist weiterhin möglich (siehe .env.example), ist aber nicht
Standard mehr.

KONTEXT (kurz)
--------------
• Deploy: GitHub Repo "Asti1982/syndiode" Branch "syndiode" → Render Web-Service
  (startCommand: python nomad_api.py). Der Python-Code enthält u.a.:
  - GET /openapi.json
  - GET /.well-known/nomad-peer-acquisition.json
  - GET /.well-known/nomad-agent.json
  - GET /.well-known/nomad-inter-agent-witness-offer.json
  usw.

• Öffentliche URL (von außen, Standard): https://syndiode.com/...

• Beobachtung: Manche Pfade liefern 200, andere 404 (z.B. /openapi.json,
  mehrere /.well-known/nomad-*.json).

• Das Muster passt zu: VOR Render sitzt eine Schicht (fast immer Cloudflare:
  DNS-Proxy, Page Rules, Worker), die NICHT ALLE Pfade an denselben
  Render-Origin durchreicht — nicht zu "Render deploy ist kaputt".

DIagnose: RENDER GRÜN, ABER /openapi.json = 404 UND /health „KURZ“?
--------------------------------------------------------------------
Prüfe die **HTTP-Antwort-Header** (Browser DevTools → Network, oder curl -I):

  Server: cloudflare
  x-nomad-edge: public-api-only
  x-render-origin-server: NomadEdge/…

Wenn diese Header da sind, geht der Traffic **nicht** direkt zu dem Prozess,
den Render aus **nomad_api.py** startet, sondern zu einem **eigenen
NomadEdge**-Dienst (z.B. Cloudflare Worker / gebündelte Route). Dann kann
Render „Deploy live“ sein und trotzdem **/openapi.json** von außen 404 liefern.

Einfachste Abhilfe (wieder „einfacher“):
  • DNS-Eintrag für **syndiode.com** (und ggf. **www**) auf den Render-Host:
    **Proxy aus** (graue Wolke „DNS only“), **oder**
  • Worker / NomadEdge so umbauen, dass **alle** Pfade 1:1 an denselben
    Render-Web-Service durchgereicht werden (kein „public-api-only“-Subset).

Ohne diese Schicht: **curl -I https://…onrender.com/health** sollte **keinen**
Header `x-nomad-edge` zeigen und **Server** typischerweise **Render** / kein CF.

WICHTIG ZU DEINEM "CLOUDFLARE API CODE"
----------------------------------------
Im Nomad-Repo / in .env.example taucht oft auf:

  CLOUDFLARE_ACCOUNT_ID
  CLOUDFLARE_API_TOKEN

Diese Variablen werden im Projekt primär für **Cloudflare Workers AI** (Inferenz/
Compute-Lane) genutzt — siehe compute_probe / infra_scout. Das ist **kein**
automatischer Ersatz dafür, Worker-Routen oder Zone-Regeln für syndiode.com
zu patchen.

Willst du per API **DNS/Worker/Routes** ändern, brauchst du ein **separates**
API-Token mit passenden Rechten (z.B. Zone DNS bearbeiten, Workers Scripts,
Route-Mappings) — oder du änderst es **manuell im Cloudflare Dashboard** (s.u.).


================================================================================
TEIL A — CLOUDFLARE DASHBOARD (ohne API): Edge so, dass API-Pfade durchgehen
================================================================================
Wo: https://dash.cloudflare.com → Zone **syndiode.com**

Ziel: Jeder Request an **https://syndiode.com** (bzw. euer gewähltes Muster), der
Nomad erreichen soll, soll den **gleichen** Backend-Origin erreichen wie die
Pfade, die heute schon 200 liefern — idealerweise **ohne** enge Pfad-Whitelist.

Schritte (je nach deinem Setup eines oder mehrere):

  A1) Workers & Pages → **Workers** / **Worker Routes**
      Prüfe, ob ein Worker an **syndiode.com** hängt und **nur ausgewählte**
      Pfade weiterleitet (Whitelist mit `available` + `not_found`).
      → Lösung: **Alle** relevanten API-Pfade an Render durchlassen, oder
         Worker so umbauen, dass **alles** unter dem Host (außer expliziter
         Blockliste) zum Origin geht.

  A2) Rules → **Transform Rules** / **Redirect Rules** / **Bulk Redirects**
      Suche nach Regeln, die nur bestimmte Pfade zulassen.
      → Lösung: Regeln lockern oder explizit z.B. /openapi.json und
         /.well-known/nomad-*.json freigeben.

  A3) Caching
      Falls aggressive Cache-Regeln 404 ausliefern: für /.well-known/* und
      /openapi.json **Bypass** oder kürzere TTL.

  A4) Origin / Proxy-Ziel
      Stelle sicher, dass das Ziel **derselbe Render-Service** ist.

Nach jeder Änderung: 2–5 Minuten warten, dann Schnelltests (PowerShell oder curl):

  curl -sS -o NUL -w "%%{http_code}" https://syndiode.com/openapi.json
  curl -sS -o NUL -w "%%{http_code}" https://syndiode.com/.well-known/nomad-peer-acquisition.json

Erwartung: 200 und JSON-Body vom Nomad-API-Prozess.

Falls du **bewusst** /nomad nutzt: dieselben Pfade mit Prefix testen, z.B.
  https://syndiode.com/nomad/openapi.json
und in Render **NOMAD_PUBLIC_API_URL=https://syndiode.com/nomad** setzen.


================================================================================
TEIL B — RENDER (Deploy & Diagnose, optional API)
================================================================================
Wo: https://dashboard.render.com → Blueprint/Service aus Repo syndiode

In deiner lokalen .env (NICHT committen):

  RENDER_API_KEY=rnd_...          (oder Alias NOMAD_RENDER_API_KEY=rnd_...)
  NOMAD_RENDER_SERVICE_ID=srv_...   (für gezielten Deploy-Trigger)
  NOMAD_RENDER_OWNER_ID=...       (falls mehrere Workspaces)

Lokal prüfen (lädt .env via python-dotenv, wenn du im Repo-Root bist):

  python main.py --cli render --json

Im Code macht RenderHostingProbe u.a. HTTP-GETs gegen NOMAD_PUBLIC_API_URL
(standardmäßig https://syndiode.com) und listet Render-Services — siehe
render_hosting.py.

Deploy manuell anstoßen (nur wenn du willst): Render Dashboard → Service →
"Manual Deploy" ODER über Nomad/Render-API mit expliziter Freigabe (siehe
render_hosting.py trigger_deploy — braucht u.a. service id).

Start Command (wichtig): In den Service-Settings **python nomad_api.py** setzen.
Falls Render stattdessen **python main.py** ohne Argumente ausführt: Im Repo
startet **main.py** dann automatisch die HTTP-API im Vordergrund (RENDER=true).
**Exit-Code 2** kommt meist von falscher CLI (**argparse**); Repo-Root enthält
zudem eine **Procfile**-Zeile `web: python nomad_api.py` als Referenz.


================================================================================
TEIL C — .env ÜBERSICHT (Namen, nicht die Werte)
================================================================================
Für **syndiode.com** öffentlich erreichbar (Nomad selbst, Standard ohne Prefix):

  NOMAD_PUBLIC_API_URL=https://syndiode.com
  NOMAD_COLLABORATION_HOME_URL=https://syndiode.com

Optional weiterhin mit Pfad-Prefix (Edge + Nomad müssen übereinstimmen):

  NOMAD_PUBLIC_API_URL=https://syndiode.com/nomad
  NOMAD_COLLABORATION_HOME_URL=https://syndiode.com/nomad

Für **Render-API** (Hosting-Checks, optional Deploy):

  RENDER_API_KEY=...              oder NOMAD_RENDER_API_KEY=...
  NOMAD_RENDER_SERVICE_ID=srv_...
  NOMAD_RENDER_OWNER_ID=...       (falls nötig)

Für **Cloudflare Workers AI** (LLM-Inferenz — NICHT dasselbe wie Edge-Routing):

  CLOUDFLARE_ACCOUNT_ID=...
  CLOUDFLARE_API_TOKEN=...        (Workers AI Berechtigungen, nicht automatisch Zone-Edit)


================================================================================
TEIL D — WENN DU PER CLOUDFLARE-API ROUTEN AUTOMATISIEREN WILLST (fortgeschritten)
================================================================================
1) Neues API-Token unter https://dash.cloudflare.com/profile/api-tokens
   mit mindestens:
   - Zone: Read (syndiode.com)
   - Workers Scripts: Edit (falls Worker geändert werden soll)
   - oder nur DNS, wenn das Problem ein A/CNAME-Record ist

2) Token **nicht** mit dem Workers-AI-Inferenz-Token vermischen — getrennt halten,
   z.B. CLOUDFLARE_EDGE_API_TOKEN (eigenen Namen wählen; Nomad nutzt ihn dann
   erst, wenn ihr dafür ein Skript/Tool einführt — im Standard-Repo ist das
   nicht eingebaut).

3) Cloudflare REST-Doku: https://developers.cloudflare.com/api/
